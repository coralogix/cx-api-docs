# cx-api-docs — developer and agent reference

This Mintlify repo publishes Coralogix API docs from OpenAPI specs. This file describes the build system, ownership rules, and workflows for both humans and AI agents.

---

## What lives here

| Path | What it is | Who/what owns it |
|---|---|---|
| `openapi_v3.yaml` / `openapi_v4.yaml` / `openapi_v5.yaml` | OpenAPI specs — source of truth for all API operations | Auto-synced from openapi-facade (CI workflow) |
| `api-reference/{v3,v4,v5}/{service}/` | Per-operation `.mdx` pages | Auto-generated by `mintlify-scrape` — **never edit manually** |
| `service-overviews/{service}-overview.mdx` | Human-authored overview page for each service | Human-owned |
| `introduction-v{3,4,5}.mdx` | Version landing pages with the service table | Human-owned |
| `build_navigation_file.py` | Copies overviews into `api-reference/`, regenerates `docs.json` | Checked-in script |
| `docs.json` | Mintlify navigation config | Auto-generated by `build_navigation_file.py` — **never edit manually** |

---

## Version policy

**v5 is the only active version.** All new APIs and services go into v5 only.

**v3 and v4 are frozen for backward compatibility.** Do not add new service overviews, new entries to `V3_SERVICES`/`V4_SERVICES`, or new rows to `introduction-v3.mdx`/`introduction-v4.mdx`. The only valid changes to those versions are bug fixes to existing content.

---

## What `make build` does

\`\`\`
mintlify-scrape openapi-file openapi_v{3,4,5}.yaml -o api-reference/v{3,4,5}
python3 build_navigation_file.py
\`\`\`

Step by step:

1. **\`mintlify-scrape\`** reads each OpenAPI spec and writes one `.mdx` file per operation into `api-reference/{version}/{service-name}/`. This wipes and regenerates that tree entirely.
2. **\`build_navigation_file.py\`** does two things:
   - Copies `service-overviews/{service}-overview.mdx` → `api-reference/{version}/{service}/overview.mdx` for every version that has that service directory.
   - Rewrites `docs.json` with the full navigation structure, based on whichever services are present on disk *and* listed in `V{3,4,5}_SERVICES` in the script.

### What `make build` does NOT do

- It does **not** touch `introduction-v{3,4,5}.mdx`. Those are never regenerated.
- It does **not** add a service to the navigation just because it appears in the OpenAPI spec. The service must also be present in the `V{3,4,5}_SERVICES` dict in `build_navigation_file.py`.
- It does **not** create or update `service-overviews/` files. Those are written by a human.
- It does **not** update the landing-page service table in `introduction-v5.mdx`. That table is maintained manually.

---

## Hidden services — scraped but not shown

`mintlify-scrape` generates directories for every service in the spec, but `build_navigation_file.py` silently skips any directory not in the `V{version}_SERVICES` dict. The following v5 services are currently generated on disk but **excluded from the nav**:

- `api-keys-admin-service`
- `cases-notification-service`
- `cases-service`
- `users-management-service`

There are also a handful of loose `.mdx` files at the `api-reference/v5/` root (operations mintlify-scrape couldn't place in a service directory) that are similarly unreachable.

To find all hidden services:

\`\`\`bash
comm -23 \
  <(ls api-reference/v5/ | sort) \
  <(python3 -c "from build_navigation_file import V5_SERVICES; [print(k) for k in sorted(V5_SERVICES)]")
\`\`\`

## Adding an existing API to docs

An API is "existing" if its operations already appear in the OpenAPI spec (i.e., `make build` already generated files for it). All work targets **v5 only**.

1. **Register the service in the v5 nav** — add an entry to `V5_SERVICES` in `build_navigation_file.py`:
   \`\`\`python
   'cases-service': 'Cases Service',
   \`\`\`
2. **Write a service overview** — create `service-overviews/cases-service-overview.mdx`. Follow the pattern of existing overviews (title, auth/permissions table, error codes, protocol docs).
3. **Add it to the intro page** — open `introduction-v5.mdx` and add a row to the API Overview table.
4. **Run `make build`** to copy the overview into place and regenerate `docs.json`.
5. **Run `mintlify dev`** to verify it renders correctly.

Do all source edits in a single commit, then commit the generated `api-reference/` and `docs.json` changes together with them.

---

## Adding a brand-new API to docs

A "new" API is one whose operations do not yet exist in the OpenAPI spec. All work targets **v5 only**.

1. **Get the spec updated** — the OpenAPI specs are auto-synced from the openapi-facade repository (see the CI workflow). Either land the spec change there and wait for the sync, or temporarily add operations to the local yaml for preview purposes.
2. **Run `make build`** — this generates the per-operation `.mdx` files.
3. **Follow the "existing API" steps above** (register in `V5_SERVICES`, write overview, update `introduction-v5.mdx`).

---

## What must never be edited

- **`api-reference/`** — fully regenerated by `mintlify-scrape` on every `make build`. Any manual edits will be overwritten.
- **`docs.json`** — regenerated by `build_navigation_file.py`. Edit the script or the source files instead.
- **`openapi_v{3,4,5}.yaml`** — auto-synced from openapi-facade. Edits here will be clobbered on the next sync.
- **`V3_SERVICES` / `V4_SERVICES`** in `build_navigation_file.py` — those dicts are frozen. New services go into `V5_SERVICES` only.
- **`introduction-v3.mdx` / `introduction-v4.mdx`** — frozen; do not add new services to those pages.

---

## What agents own

| Task | File(s) to change |
|---|---|
| Add/remove a service from the v5 nav | `build_navigation_file.py` → `V5_SERVICES` |
| Add/update a service overview | `service-overviews/{service}-overview.mdx` |
| Add/update the v5 landing page table | `introduction-v5.mdx` |

---

## Verifying changes

After `make build`, check:
- The service directory exists under `api-reference/v5/{service}/`
- `overview.mdx` is present in that directory (confirms the overview copy step ran)
- `docs.json` contains the service under the v5 navigation groups
- `mintlify dev` shows the service in the sidebar and the overview page loads

---

## Overview file format

Follow the pattern of existing files in `service-overviews/`. Minimum sections:

\`\`\`markdown
# {Service display name} overview

{One paragraph describing what the service does.}

## Authentication and permissions

{API key requirements and permission table.}

## Common error response codes

{Table of status codes.}
\`\`\`

Do not add version-specific content to overview files — they are copied verbatim into all versions that have the service directory.

## Running locally

\`\`\`bash
make build      # regenerate api-reference/ and docs.json
mintlify dev    # local preview server
\`\`\`

`make clean` removes `api-reference/` entirely. Run `make build` afterward to regenerate.
